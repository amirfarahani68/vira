# Risk Register

This register identifies potential risks associated with Phase 1 of the Vira attendance platform and outlines mitigation strategies.  Risks are grouped by category and assessed qualitatively in terms of likelihood (Low/Medium/High) and impact (Low/Medium/High).

| Risk | Description | Likelihood | Impact | Mitigation |
|-----|-------------|-----------|--------|-----------|
| **Geolocation accuracy** | Attendance may rely on geolocation to ensure employees are at the branch when clocking in/out.  Mobile browsers can report inaccurate or unavailable coordinates, especially indoors, leading to false negatives. | Medium | Medium | Make geolocation optional in Phase 1.  Provide fallback methods (QR and manual).  Log geolocation but do not block attendance when GPS fails.  Consider using Wi‑Fi SSIDs or IP ranges for location verification in later phases. |
| **QR code abuse** | Rotating QR codes can be screenshotted or shared.  Without server‑side validation, a technician could scan an old QR or share it with a colleague. | High | High | Generate QR tokens on the server and store them in Redis with a short TTL (e.g., 60 seconds).  Validate tokens on scan and associate them with branch and shift.  Track device IDs or session IDs to detect multiple scans from the same user. |
| **Browser limitations** | Different browsers (especially mobile Safari vs. Chrome) may handle camera access, localStorage and offline behaviour differently, leading to inconsistent experiences. | Medium | Medium | Test across major browsers.  Use standard APIs for camera/scanning.  Do not rely on localStorage for critical data; persist everything in the backend.  Provide progressive enhancement for offline scenarios. |
| **Authentication & session security** | Implementing JWT or session cookies must be done carefully to prevent token theft, CSRF and replay attacks. | Medium | High | Use secure HTTP‑only cookies for refresh tokens.  Implement CSRF protection on state‑changing requests.  Store password hashes using a strong algorithm (bcrypt or Argon2).  Regularly rotate JWT secrets. |
| **RBAC complexity** | Mistakes in role permissions could allow unauthorised access to data (e.g., a technician viewing all branches). | Low | High | Define permissions centrally and test them thoroughly.  Use NestJS guards to enforce RBAC at controller and route levels.  Include integration tests for each role’s allowed and denied actions. |
| **Multi‑branch data leaks** | Without proper scoping, data queries may return information from other branches.  For example, a supervisor might see attendance records from a different branch. | Medium | High | Always include `branchId` filters in queries.  Implement unit tests to ensure data is scoped correctly.  Use multi‑tenant patterns (e.g., global query filters) in the ORM. |
| **Audit logging omissions** | Failing to log critical actions (approvals, edits, deletions) compromises compliance and hinders troubleshooting. | Medium | Medium | Implement a global interceptor in NestJS that logs specified actions.  Review logs regularly.  Make the AuditLog entity append‑only to prevent tampering. |
| **Persian calendar handling** | Using Gregorian calendars for date/time fields could confuse users who operate on the Persian calendar.  Incorrect date conversions may cause shift or leave errors. | Medium | Medium | Use a date library (e.g., `jalali-moment` or `dayjs` with Jalali plugin) in the frontend to display and input dates in the Persian calendar.  Store dates as ISO (Gregorian) in the database and convert on display.  Educate users about the dual calendar system. |
| **Liara deployment misconfiguration** | Misconfigured build/start commands or environment variables could cause deployment failures. | Low | High | Prepare `liara.json` with correct build (`next build`, `nest build`) and start commands.  Use `.env.example` to document all variables.  Test deployments on a staging environment before production. |
| **Infrastructure scaling & cost** | As usage grows, the single‑instance deployment may become a bottleneck.  Without proper caching and scaling, performance will degrade. | Low | Medium | Use Liara’s scaling options to increase resources.  Employ Redis caching for expensive queries.  Monitor performance and plan horizontal scaling if needed. |
| **Data integrity** | Concurrent updates (e.g., two supervisors editing shifts simultaneously) can cause race conditions.  | Medium | Medium | Use optimistic locking or transactions in the ORM.  Validate data on the server.  Provide UI warnings when data has changed since retrieval. |
| **Compliance and data privacy** | Storing employee attendance and leave data must comply with labour laws and data protection regulations. | Low | High | Keep personal data minimal.  Encrypt sensitive fields at rest.  Provide access logs and allow users to view their own data.  Consult legal counsel for regional compliance. |

## Summary

The risks above represent the most significant technical and operational challenges for Phase 1.  By anticipating them and implementing the mitigation strategies, the Vira team can increase the likelihood of a successful rollout and avoid critical issues during production.
